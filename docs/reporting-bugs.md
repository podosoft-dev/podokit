# Reporting a bug

Found something wrong in a project generated by PodoKit, in the `podo` CLI, or in
the templates/modules themselves? File it against this repository. A good report
lets us reproduce and fix the issue fast — often it is the difference between a
same-day fix and a back-and-forth that never converges.

Bugs go to the public tracker: **https://github.com/podosoft-dev/podokit/issues**.
For a **security vulnerability**, do not open a public issue — follow
[SECURITY.md](../SECURITY.md) instead.

## Pick a path

| You are… | Do this |
| --- | --- |
| A person, in a browser | Open a [**Bug report**](https://github.com/podosoft-dev/podokit/issues/new?template=bug_report.yml) issue and fill the form. |
| A person, at a terminal | Write a report from the [template](#report-template) and file it with [`gh`](#file-with-the-github-cli). |
| An AI coding agent (Claude, Codex, …) | Follow [Filing from an AI agent](#filing-from-an-ai-agent) — generate the report, then run one `gh` command. |

Either way, aim for the same content. The form and the template below ask for the
same things; the template just carries more room for evidence and a suspected root
cause, which speeds up triage.

## What makes a report actionable

- **Reproduction that starts from zero.** The exact commands, from `npx @podosoft/podokit create …`
  onward — including the template, the modules added (`podo add …`), and how you ran it
  (host `npm run dev` vs. containerized `compose.dev.yaml`, and any non-default `TRAEFIK_PORT`/ports).
- **Expected vs. actual**, stated separately.
- **Evidence, not adjectives.** Paste the failing log line, the browser console/network error,
  the stack trace, or a screenshot. "Doesn't work" is not reproducible; `WebSocket to
  'ws://app.localhost/…' failed` is.
- **Environment.** OS, Node version, package manager + version, and PodoKit/CLI version
  (`podo --version`, or the `@podosoft/podokit*` versions in your lockfile).
- **A suspected root cause or fix, if you have one** — optional, but it turns a report into a
  head start. Point at the file and line if you can.

## Report template

Copy this into a `REPORT-<slug>.md` file (or straight into the issue body). Keep the
headings — they map directly onto how we triage. Delete the italic hints.

```markdown
# <one-line summary of the bug>

## Summary
*What is broken, in one or two sentences.*

## Environment
- PodoKit / CLI version: *e.g. @podosoft/podokit 0.7.0 (`podo --version`)*
- Template: *fullstack-nest-svelte | todo | base*
- Modules added: *e.g. auth, admin-dashboard — or "none"*
- Run mode: *host `npm run dev` | containerized `compose.dev.yaml` (note any custom TRAEFIK_PORT/ports)*
- OS / Node / package manager: *e.g. macOS 15 / Node 22.11 / npm 10*

## Steps to reproduce
1. *exact command, starting from `npx @podosoft/podokit create …`*
2. …

## Expected
*What should happen.*

## Actual
*What actually happens.*

## Evidence
*Failing log lines, browser console/network errors, stack traces, screenshots.*

## Suspected root cause / suggested fix (optional)
*File + line if known, and what you think should change.*
```

## File with the GitHub CLI

Write the report to a file, then:

```bash
gh issue create \
  --repo podosoft-dev/podokit \
  --title "<one-line summary>" \
  --label bug \
  --body-file REPORT-<slug>.md
```

`gh` prints the new issue URL. (Run `gh auth login` once if you have not.)

## Filing from an AI agent

PodoKit is built to be driven by AI coding agents, and reporting is no exception.
If you are an agent (Claude Code, Codex, Cursor, …) that hit a bug while working in
a generated project, you can file it directly:

1. Write the bug up as `REPORT-<slug>.md` using the [template](#report-template)
   above. Fill **Evidence** from what you actually observed (console/network logs,
   command output) — do not invent values.
2. Include the reproduction from a clean `create`, the run mode, and — if you can
   trace it — the suspected file and line under **Suspected root cause**.
3. File it:

   ```bash
   gh issue create --repo podosoft-dev/podokit \
     --title "<one-line summary>" --label bug --body-file REPORT-<slug>.md
   ```

4. Report the resulting issue URL back to the user. Do **not** file the same bug
   twice — search first: `gh issue list --repo podosoft-dev/podokit --search "<keywords>"`.

Never include secrets, tokens, private hostnames/IPs, or proprietary code in a
report — redact them first.
