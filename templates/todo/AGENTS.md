## Setup & commands

```bash
npm install
cp .env.example .env
docker compose -f infra/docker/docker-compose.yml up -d   # PostgreSQL (+ Redis)
npm run dev        # api on http://localhost:5002, web on http://localhost:5001
```

- **Ports: web = 5001, api = 5002.** The web app reaches the API only through its
  SvelteKit server proxy (`/api/*`), never with a direct browser `fetch`.
- Build: `npm run build`  ¬∑  Type-check/lint: `npm run lint`  ¬∑  Unit tests: `npm test`
- Per workspace: `npm run build -w {{projectName}}-api` / `-w {{projectName}}-web`.
- DB migrations (TypeORM): `npm run migration:run -w {{projectName}}-api`.
- e2e (Playwright, ships in `tests/`): `npm run test:e2e`.

**Always** verify a change by building the affected workspace (and `svelte-check`
for web) before considering it done. UI changes need a Playwright test; API
changes need an e2e/unit test.

## Code style

### TypeScript (both apps)
- `strict` mode. **No `any`** ‚Äî use `unknown` + narrowing. **No `@ts-ignore`/`@ts-expect-error`** and no `eslint-disable`.
- Explicit function return types. DTOs use `class-validator` decorators.
- Keep `noUnusedLocals`, `noUnusedParameters`, `noImplicitReturns`, `noUncheckedIndexedAccess`.

### Svelte 5 (web)
- **Runes only**: `$state()`, `$derived()`, `$effect()`, `$props()`. **Never** use `$:` (deprecated).
- Reactive stores go in `.svelte.ts` files; don't destructure a getter-exported rune (it loses reactivity) ‚Äî wrap in `$derived`.

### UI ‚Äî Tailwind + shadcn-svelte (web)
- Prefer **shadcn-svelte components** over raw HTML (`<button>`, `<input>`, `<select>`, ‚Ä¶).
- **Do not edit `src/lib/components/ui/**`** (the shadcn primitives). Customize in a separate wrapper.
- Tailwind v4 (CSS-first, no `tailwind.config`). Use semantic classes: `bg-background`, `text-foreground`, `border-border`. Dark mode via `mode-watcher`. Responsive (`sm:`/`md:`/`lg:`).

### i18n & errors
- **Every user-facing string goes through i18n** (`src/lib/i18n/messages.ts`) ‚Äî no hardcoded text. Add keys to **all** locales at once.
- The backend returns a stable **error envelope**: `{ success: false, error: { code, message, statusCode, path, timestamp, details? } }`. Throw `AppException("SOME_CODE", ...)` on the backend; **branch on `error.code`, not the message,** on the frontend.
- Logging is structured; no emojis/icons. Start messages with an action verb.

## PodoKit tooling & file ownership

This project is managed by the `podo` CLI. Files have an ownership tier recorded
in `.podokit/files.lock` ‚Äî respect it:

- **owned** (your code: `apps/web/src/routes/**`, your components/services, `apps/api/src/app.extensions.ts`, and this `AGENTS.md`) ‚Äî edit freely. To add or override Nest providers/modules, use `app.extensions.ts` (spread into `AppModule` after the module-wired providers, so a same-token override wins) rather than editing `app.module.ts`.
- **assembled** (e.g. `apps/api/src/app.module.ts`, `auth.ts`) ‚Äî contain `// podokit:begin:‚Ä¶ / ‚Ä¶:end` fenced regions that the toolkit recomputes. Edit outside the fences only; **never hand-edit inside the fences**.
- **managed** (bootstrap/config like `main.ts`) ‚Äî the toolkit may update these; avoid gratuitous edits.

Useful commands:

- `podo add <module>` ‚Äî add a feature (auth, admin-dashboard, redis, bullmq, ‚Ä¶). It wires itself into the app; run `podo add` with no argument to list modules.
- `podo remove <module>` ‚Äî un-apply a module (inverse of add; refuses if another module needs it, keeps files you edited).
- `podo status` / `podo diff` ‚Äî see your local edits vs. what PodoKit generated.
- `podo update` ‚Äî pull in template/module improvements (3-way merges your edits; never touches owned files).
